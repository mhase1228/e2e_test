## やり取り 1：サイト構成の分析とフォーマット案の提示

### ユーザーのプロンプト

> https://hotel-example-site.takeyaqa.dev/ja/index.html の構成を分析し、これを対象とした、e2e テスト仕様書を作成してください。まずは、テスト仕様書のフォーマットを決めましょう。案を出してください。

### AIの回答

サイト構成を調査し、以下の3つのフォーマット案を提示した。

---

#### 案A: Markdown テーブル形式（シンプル）

ページ単位でセクションを分け、テストケースを表で列挙するスタンダードな形式。

```markdown
## TC-LOGIN: ログイン

| テストID    | テスト名                       | 前提条件       | 手順                            | 期待結果             | 優先度 |
| ----------- | ------------------------------ | -------------- | ------------------------------- | -------------------- | ------ |
| TC-LOGIN-01 | 正常ログイン（プレミアム会員） | 未ログイン状態 | 1. /ja/login.html を開く<br>... | マイページに遷移する | High   |
```

**向いている用途:** 読み書きが簡単、Git管理しやすい、チームへの共有が容易

---

#### 案B: BDD スタイル（Gherkin 風）

Given / When / Then でシナリオを記述。テストコードと対応しやすい形式。

```markdown
## Feature: ログイン

### Scenario: プレミアム会員が正常にログインできる
- **Given** 未ログイン状態でログインページを開いている
- **When** ichiro@example.com / password でログインする
- **Then** マイページへ遷移する
```

**向いている用途:** 非エンジニアも読める、Playwright テストコードに直接マッピングしやすい

---

#### 案C: テスト設計書形式（詳細）

テストID・優先度・種別・境界値などを明示した、フォーマルな仕様書形式。

```markdown
## TC-LOGIN-01

| 項目       | 内容                           |
| ---------- | ------------------------------ |
| テストID   | TC-LOGIN-01                    |
| テスト名   | 正常ログイン（プレミアム会員） |
| 対象ページ | /ja/login.html                 |
| テスト種別 | 正常系                         |
| 優先度     | High                           |
| 前提条件   | 未ログイン状態                 |

**手順**
1. `/ja/login.html` を開く
2. ...

**期待結果**
- マイページに遷移する
```

**向いている用途:** 品質管理・テスト進捗管理を重視する場合

---

#### 比較まとめ

|                | 案A（テーブル） | 案B（BDD） | 案C（詳細） |
| -------------- | --------------- | ---------- | ----------- |
| 記述量         | 少ない          | 中程度     | 多い        |
| 可読性         | ふつう          | 高い       | 高い        |
| コードとの対応 | ふつう          | 直接的     | 間接的      |
| 管理のしやすさ | 高い            | ふつう     | やや重い    |

## やり取り 2：テスト仕様書の作成

### ユーザーのプロンプト

> 案Cにします。では、このフォーマットで、テスト仕様書を作成してください。

### AIの回答

サイトの各ページ（ログイン・会員登録・宿泊プラン一覧・宿泊予約・宿泊予約確認・マイページ）をPlaywright MCPで実際にアクセスして調査し、案Cのフォーマットでテスト仕様書を作成した。

**調査したページ:**
- `/ja/login.html` — メールアドレス・パスワードの2フィールド
- `/ja/signup.html` — メールアドレス、パスワード（確認含む）、氏名（必須）、住所・電話番号・性別・生年月日・お知らせ（任意）
- `/ja/plans.html` — 7種類のプラン（非同期読み込み）、未ログイン/一般会員/プレミアム会員で表示が変化
- `/ja/reserve.html?plan-id=0` — 宿泊日・宿泊数・人数・追加プラン・氏名・確認のご連絡・ご要望、動的合計金額計算
- `/ja/mypage.html` — ログイン後のみアクセス可、ユーザー情報一覧表示、アイコン設定・退会ボタン（新規登録ユーザーのみ有効）

**作成したファイル:** `docs/test-spec.md`

**作成したテストケース数: 47件**
